# Технический аудит проекта автоматического формирования отчетности

Дата проверки: 14.06.2026

## 1. Общая оценка проекта

Проект представляет собой учебно-прикладное Flask-приложение с MySQL, серверными маршрутами для отчетов, шаблонов, документов, организации, пользователей, групп, прав доступа, логов и персональных настроек. Кодовая база уже содержит значимый функционал: редакторы отчетов и шаблонов, импорт DOCX/XLSX/CSV/PDF, админскую панель, SmartCaptcha, организации, группы пользователей и совместный доступ.

Основные риски до исправлений были связаны не с отсутствием функций, а с безопасностью границ: запуск по заявленной команде, отсутствие общей CSRF-защиты, сохранение HTML редакторов без серверной очистки, слабая проверка загружаемых файлов и небезопасный пример `.env.example`.

Часть рисков требует отдельной миграции данных: у отчетов, папок и черновиков нет явных полей владельца/организации, поэтому полная защита от IDOR для этих объектов не может быть корректно добавлена без согласованного изменения схемы и обратного заполнения старых данных.

- Шаблоны: базовый layout, auth templates, редакторы отчетов и шаблонов, админская панель, пользовательские настройки.

- Полная защита IDOR для отчетов/папок/черновиков требует миграции БД: `organization_id`, `owner_id`, backfill старых данных и обновление всех выборок.

## 6. Риски информационной безопасности

- `SECRET_KEY` должен быть задан в локальном `.env`; fallback допустим только для разработки.
- Локальный `.env` не проверялся на публикацию, но `.env.example` очищен.
- Загруженные аватары остаются публичными static-файлами. С учетом whitelist изображений риск снижен, но приватность аватаров не обеспечивается.
- В отчетах/папках нет серверной границы организации/владельца. Это главный оставшийся риск доступа.
- Login brute force пока не ограничен.
- Старые записи `content_html/document_html` в БД могли быть сохранены до санитайзера; при выдаче они очищаются, но желательно выполнить ручную проверку/миграционную очистку.

## Реализация экспорта отчётов

Реализован единый защищенный экспорт сформированных отчетов в три формата:

- DOCX (`application/vnd.openxmlformats-officedocument.wordprocessingml.document`);
- PDF (`application/pdf`);
- XML (`application/xml; charset=utf-8`).

Добавленный маршрут:

- `POST /reports/<report_id>/export/<format>`, где `format` может быть `docx`, `pdf` или `xml`.

Серверная логика вынесена в `app/report_export_service.py`. Все форматы используют один и тот же источник данных отчета: связанный черновик, актуальное HTML-содержимое редактора, метаданные отчета и импортированные блоки как резервный источник. HTML перед экспортом проходит через `sanitize_editor_html`, запрещенные теги и обработчики событий не попадают в файл.

Права доступа проверяются на сервере перед формированием файла:

- администратор может экспортировать отчет;
- участник может экспортировать отчет при прямом совместном доступе через `report_shares`;
- участник может экспортировать отчет при доступе через активную группу;
- запрос без авторизации получает отказ;
- неизвестный формат получает `400`, отсутствующий отчет получает `404`.

Интерфейс:

- в окно предпросмотра добавлена кнопка `Экспорт` с меню `Word (.docx)`, `PDF (.pdf)`, `XML (.xml)`;
- в редактор отчета добавлена та же кнопка экспорта;
- при экспорте кнопки блокируются, показывается состояние загрузки, скачивание выполняется без перезагрузки страницы;
- если в редакторе есть несохраненные изменения, frontend сначала вызывает сохранение редакторского состояния, дожидается успешного ответа и только затем отправляет запрос экспорта.

Библиотеки:

- `python-docx` для DOCX;
- `reportlab` для PDF;
- `beautifulsoup4` для разбора очищенного HTML;
- `pypdf` используется в автоматических тестах для проверки читаемости PDF.

Временные файлы не создаются: DOCX/PDF/XML формируются в памяти через `BytesIO`. Имена файлов формируются безопасно в виде `report_<id>_<slug>_<date>.<extension>` через транслитерацию, ограничение длины и `secure_filename`.

Журналирование:

- успешный экспорт пишет событие `report.export` в журнал действий пользователей;
- в событии сохраняются пользователь, id отчета, формат и время операции;
- содержимое документа, пароль, токены и данные сессии в журнал не записываются.

Известные ограничения:

- DOCX/PDF-конвертация поддерживает основные HTML-элементы редактора, таблицы, изображения из data URI и локального `/static`, списки, заголовки и базовые inline-стили; сложные CSS-позиционирования и нестандартные элементы редактора экспортируются best-effort.
- PDF использует найденный локальный кириллический шрифт (`Arial`/`DejaVu Sans`). На текущей Windows-среде генерация и чтение PDF проверены; на другой машине без этих шрифтов нужно установить подходящий TTF или добавить путь к нему.
- В таблице `reports` пока нет полноценного `owner_id`/`organization_id`, поэтому проверка владельца и организации ограничена имеющейся моделью: admin, прямой share и group share. Для строгой multi-tenant-защиты нужна отдельная миграция владельца/организации отчетов.

## 10. Инструкция по повторной проверке

1. Установить зависимости:
   ```powershell
   py -m pip install -r requirements.txt
   ```

2. Запустить приложение заявленной командой:
   ```powershell
   py app.py
   ```

   Альтернатива через проектный venv:
   ```powershell
   .\venv\Scripts\python.exe app.py
   ```

3. Запустить тесты:
   ```powershell
   .\venv\Scripts\python.exe -m pytest -q
   ```

4. Проверить вручную:
   - вход/выход;
   - создание отчета;
   - импорт PDF/DOCX/XLSX/CSV;
   - сохранение редактора отчета;
   - сохранение редактора шаблона;
   - админ-панель пользователей/групп/прав/логов;
   - загрузку аватара;
   - отказ при загрузке файла с неверной сигнатурой;
   - отсутствие выполнения вставленного `<script>` в редакторах.
